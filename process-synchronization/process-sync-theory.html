Process Synchronization Theory

Producer‑Consumer (Bounded Buffer)

Concept: A fixed‑size buffer shared by producers (add data) and consumers (remove data). Requires synchronization to avoid race conditions (e.g., producer overwriting unread data, consumer reading empty buffer).

Classic solution using semaphores:

  • empty – counts empty slots (initialized to buffer size)
  • full – counts filled slots (initialized to 0)
  • mutex – protects buffer access (binary semaphore)

Producer logic: wait(empty) → wait(mutex) → produce → signal(mutex) → signal(full)

Consumer logic: wait(full) → wait(mutex) → consume → signal(mutex) → signal(empty)

Sample Execution (buffer size = 3)

TimeEventBuffer State
0P0 produces item A[A, _, _]
1P1 produces item B[A, B, _]
2C0 consumes A[B, _, _]
3C1 consumes B[_, _, _]

Readers‑Writers Problem

Concept: Multiple readers can access shared data simultaneously, but writers need exclusive access. Variants: readers‑priority (may starve writers), writers‑priority, and fair (FIFO).

Solution (readers‑priority):

  • readCount – number of active readers
  • mutex – protects readCount
  • resourceMutex – ensures writer exclusivity

First reader locks resourceMutex, last reader releases it. Writers wait until readCount == 0.

Dining Philosophers

Concept: N philosophers seated around a table, each needs two forks to eat. Without proper strategy, deadlock (everyone holds one fork) or starvation can occur.

Deadlock avoidance strategies:

  • Resource Hierarchy: label forks 0..N-1, each philosopher picks up lower‑numbered fork first → breaks circular wait.
  • Asymmetric: even philosophers pick left then right, odd pick right then left.
  • Arbitrator (waiter): a central mutex allows only one philosopher to attempt picking forks at a time.

Sample Execution (Resource Hierarchy, N=5)

PhilosopherLeft ForkRight ForkAction
